1. The Fundamental Problem: The KV Cache "Memory Tax"

During autoregressive generation (predicting one token at a time), the model must look back at all previous tokens to understand context. Mathematically, the Attention Mechanism utilizes Keys (relevance identification) and Values (content information storage).

Naive Implementation vs. Fragmentation:
In a standard setup, the model re-calculates Keys and Values for every preceding token at every step, which is computationally prohibitive. Developers solve this using a KV Cache, storing vectors in VRAM after the first calculation.

The Fragmentation Crisis: Standard systems allocate a "maximum possible length" (e.g., 2048 tokens) per user. If a user writes a 10-token prompt, 99% of that memory is idle. This Internal Fragmentation is the primary reason standard GPUs fail to handle high concurrency.

2. The PagedAttention Revolution

Inspired by Operating System virtual memory (paging), vLLM avoids forcing the GPU to find large, continuous "parking spaces" for sequences. Instead, it breaks the cache into atomic, fixed-size Blocks.

Anatomy of a Block

  • Token Capacity: Atomic units holding data for exactly 16 tokens.
  • Model Dimensions: For TinyLlama, each token utilizes Grouped Query Attention (GQA) with 4 Key heads and 4 Value heads. Each head has a dimension of 64.
  • The "Square" (128 Bytes): At FP16 (2 bytes per number), one 64-dim head vector takes exactly 128 bytes.
  • Total Block Footprint: 8 heads/token = 1,024 bytes (1KB). 16 tokens * 1KB = 16 KB per layer.

3. Autoregressive Lifecycle: Prefill vs. Decoding

Step 1: Arrival and Tokenization

Prompt text is converted to Token IDs via CUDA cores. The system assigns the request to the Waiting Queue.

Step 2: Prefilling (The Digestive Phase)

The model processes the entire prompt at once to generate the initial KV cache. The scheduler checks the Token Budget (typically 2048). If under budget, the batch is processed in one iteration, and the first set of KV Blocks is pulled from the Free Block Queue.

Step 3: Decoding (The Generation Loop)

Once the first token is predicted, the request moves to the Running Queue. The model generates one token at a time, retrieving previous 16-token blocks from HBM3e and appending new data to the active block.

4. Continuous Batching & The Page Table

Traditional "Static Batching" wastes time waiting for the longest request in a batch to finish (the "Bus Analogy"). Continuous Batching acts at the iteration level, allowing new requests to "hop on" as soon as a block is freed.

// vLLM Logical Mapping via CPU Page Table
Sequence_A: [0-15] -> Phys_Addr: 0x7F4A (Block #500)
Sequence_A: [16-31] -> Phys_Addr: 0x1A2B (Block #12)

This separation of the Logical Map (CPU) from the Physical Data (GPU) enables:

  • Prefix Caching: Storing common system prompts once and sharing them across multiple users.
  • Copy-on-Write: Effortlessly branching conversations without duplicating initial memory blocks.
