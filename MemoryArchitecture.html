1. The KV Cache Bottleneck: The "Memory Tax"

To understand vLLM, one must first identify the "memory tax" inherent to Autoregressive Inference. Unlike standard classification, an LLM must maintain a constant "contextual memory" of every word in a sequence to predict the next token accurately.

The Crisis of Internal Fragmentation:
Standard inference engines (like naive PyTorch wrappers) pre-allocate a fixed, contiguous memory buffer for every user (e.g., 2048 tokens). If a user provides a short 10-token prompt, **99.5%** of that expensive VRAM sits idle. Because LLMs process data in batches, this inefficiency limits a 141GB H200 to only a handful of concurrent users.

Mathematically: The Attention Mechanism requires two specific vector sets per token per layer: Keys (addressing vectors) and Values (content vectors). Storing these vectors after their initial calculation prevents $O(N^2)$ computational explosion but introduces a massive storage overhead.

2. The Paged Attention Revolution

Inspired by Virtual Memory Management in modern Operating Systems, vLLM abstracts physical memory into Blocks. This allows the GPU to utilize non-contiguous VRAM "holes," effectively eliminating external fragmentation.

Anatomy of a KV Block

In the vLLM architecture, a **Block** is the atomic, indivisible unit of cache memory.

  • Token Capacity: Precisely 16 tokens per block.
  • Model Dimensions: For TinyLlama (Grouped Query Attention), each token generates 4 Key heads and 4 Value heads. Each head is a vector of 64 dimensions.
  • The "Square" (128 Bytes): Using 16-bit precision (FP16), each number takes 2 bytes. Therefore, one 64-dim head vector takes exactly **128 bytes** (64 * 2).
  • Total Block Footprint: 8 heads/token (4K + 4V) = 1,024 bytes (1 KB). Multiply by 16 tokens = **16 KB per layer, per block.**

3. Autoregressive Lifecycle: Prefill vs. Decoding

Step 1: Arrival & Tokenization

The system converts raw text to Token IDs. The request is queued in the Waiting Queue until HBM3e block availability is confirmed by the Block Manager.

Step 2: Prefilling (The Digestive Phase)

The GPU processes the entire input prompt in parallel (Batch GEMM) to build the initial KV cache. The scheduler allocates initial blocks from the Free Block Queue. If the prompt length > 16, multiple blocks are assigned via pointers.

Step 3: Decoding (The Autoregressive Loop)

The request moves to the Running Queue. The GPU predicts a single new token, retrieves the existing KV Blocks from VRAM, calculates the attention scores, and appends the new KV row to the current active block.

4. Continuous Batching & The CPU Page Table

Standard "Static Batching" waits for all requests in a batch to finish (the "Bus Analogy"). Continuous Batching acts at the iteration level—as soon as one request generates its last token, its blocks are wiped and recycled, allowing a new request to "hop on" mid-calculation.

// vLLM Logical to Physical Mapping
// Sequence: "The animal didn't cross the..."
Sequence_Alpha [Slot 0-15] -> HBM3e_Phys_Addr: 0x7A4F (Block #500)
Sequence_Alpha [Slot 16-31] -> HBM3e_Phys_Addr: 0x1B2C (Block #12)

This CPU Page Table architecture allows for Prefix Caching. If 1,000 users ask a question about the same 50-page PDF, vLLM stores that PDF's KV blocks **once** and points all 1,000 users to those same physical addresses, saving gigabytes of memory.

5. Computational Efficiency & Hardware Offloading

The vLLM engine manages two distinct resource pools to prevent system crashes (OOM):

// Memory Pool Partitioning (H200 Example)
CUDA_BLOCK_POOL ≈ 32,357 (Primary High-Speed KV Pool in VRAM)
CPU_BLOCK_POOL ≈ 11,915 (Asynchronous Swap Pool in System RAM)

When VRAM hits a critical threshold, the Block Manager initiates **Preemption**. It "swaps" older, less-active blocks to the 4GB CPU Swap space via DMA engines, bringing them back only when the user’s sequence resumes. This fluidity allows one H200 to perform the workload previously requiring an 8-GPU cluster.
