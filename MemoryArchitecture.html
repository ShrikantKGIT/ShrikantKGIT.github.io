1. KV Cache: The Memory Tax Crisis

Autoregressive inference demands contextual memory for every token. The Attention mechanism requires Keys (query matching) and Values (content storage) for all prior tokens.

Fragmentation Catastrophe:
Naive engines pre-allocate max_seq_len (e.g., 2048 tokens) per user. A 10-token prompt wastes 99.5% VRAM. On H200's 141GB, this limits to ~5-8 concurrent users.[web:2][web:9]
Attention(Q,K,V) = softmax(QKT/√d)V
// KV Cache avoids recomputing K,V → O(n²) explosion → but eats VRAM

Llama3.1-70B Example:

80 layers × 8 KV heads × 128 dim × 8192 seq × batch=32 × FP16 = 80GB KV cache. Without paging: OOM city.[code_file:1]

2. PagedAttention: OS-Inspired Revolution

vLLM breaks KV cache into fixed 16-token Blocks, enabling non-contiguous allocation like OS virtual memory.[web:6]

  • Capacity: 16 tokens/block
  • Per token: 4K heads + 4V heads × 64 dim × FP16 (2B) = 1,024B

3. Lifecycle: Prefill → Decode

① Arrival/Tokenization

Text → Token IDs. Waiting Queue until Block Manager confirms availability.

② Prefill Phase

Parallel GEMM computes initial KV for entire prompt. Allocates blocks from FreeBlockQueue. Token budget: typically 2048.[web:7]

③ Decode Loop

Running Queue. Predict 1 token → fetch KV blocks → append new row. Continuous batching swaps finishers instantly.

# Example: Block allocation trace
prompt="The quick brown fox" # 5 tokens → Block #500 (slots 0-15)
decode_step_1 → append to #500 (slot 5)
decode_step_20 → allocate #501 (slots 16-31)
EOS → recycle #500,501 → FreeQueue

4. Continuous Batching + Prefix Caching

Static batching = bus waits for slowest passenger. vLLM batches at iteration level.[web:3][web:8]

# CPU Page Table Mapping
UserA[0:16] → PhysBlock#500 (0x7A4F)
UserA[16:32] → PhysBlock#12 (0x1B2C)
UserB[0:16] → PhysBlock#500 (SHARED PREFIX!)

Real-World: RAG Pipeline

100 users query same 50-page PDF → 1x KV storage for prefix. Each branches uniquely post-prompt. Saves 90%+ VRAM.[web:3][web:8]
