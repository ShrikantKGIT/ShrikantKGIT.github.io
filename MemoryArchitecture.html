GPT-OSS-20B Memory & Throughput Blueprint

White-background technical explainer focused on model size, KV cache, and throughput

1. What GPT-OSS-20B Is

gpt-oss-20b is described as a medium-sized open-weight model for low-latency, local, or specialized use cases, with 21B total parameters and 3.6B active parameters. It also has a long context window of 128k tokens, with some references using 131,072 tokens as the precise value. [web:14][web:11]

This combination matters because the model size determines how hard it is to fit weights in GPU memory, while the context length determines how much KV cache memory you may need during inference. [web:12][web:11]

Design takeaway: the weights are only part of the memory story. For long prompts, KV cache can become the dominant memory consumer even when the model weights fit comfortably. [web:12][web:4]

2. Weight Memory Calculation

The simplest estimate is:

weight_memory = parameter_count × bytes_per_parameter

Using 21B total parameters, the raw weight sizes are:

Precision Bytes per parameter Raw weight memory
FP16 2 42.0 GB
INT8 1 21.0 GB
INT4 0.5 10.5 GB
Example: if you run the model in 4-bit form, the raw weights can be around 10.5 GB before runtime overhead. That is why the model is described as fitting in roughly 16 GB of VRAM with optimized quantization and runtime tricks. [web:12][web:11]

Because gpt-oss-20b also has only 3.6B active parameters, the effective compute footprint can be much lower than the raw 21B-parameter count suggests. [web:14]

3. KV Cache Calculation

KV cache memory scales with layers, KV heads, head dimension, context length, and precision. A compact formula is:

KV bytes per token = effective_layers × 2 × kv_heads × head_dim × bytes_per_element

For the model details commonly cited for gpt-oss-style architectures, a useful estimate is 8 KV heads, 64 head dimension, FP16 storage, and 18 effective layers if half of 36 layers are treated as sliding-window layers. That gives:

per_token_KV = 18 × 2 × 8 × 64 × 2 bytes = 36,864 bytes ≈ 36 KB/token
Context length Approx KV memory
4,096 tokens ~0.14 GB
32,768 tokens ~1.13 GB
131,072 tokens ~4.5 GB
Important: these are prompt-length-only estimates. Real serving also needs room for activations, allocator overhead, workspace buffers, and fragmentation. Long-context plus high concurrency can make KV cache much larger than the weight footprint. [web:12][web:4]

4. Why vLLM Helps

vLLM reduces memory waste by storing KV cache in fixed-size blocks rather than reserving one giant contiguous region per request. That lets the engine pack many sequences into fragmented GPU memory more efficiently. [web:6][web:3]

Block model

A practical block example is 16 tokens per block. If each token uses about 36 KB of KV cache in the estimate above, then one 16-token block can represent roughly:

16 × 36 KB ≈ 576 KB per block

This makes the allocation unit explicit and makes it easier to recycle memory as requests finish. [web:6][web:8]

5. Throughput Expectations

Throughput depends on prompt length, output length, batching, cache reuse, and whether you are compute-bound or memory-bound. Reported numbers for gpt-oss-20b include around 263 tokens/s in one benchmark and much higher aggregate throughput under concurrency on dedicated hardware. [web:13][web:18][web:15]

Scenario Interpretation Example throughput
Single-stream interactive use Latency-sensitive, small batch ~250 to 300 tokens/s
Moderate batching Mixed workload with several active requests ~2,000 tokens/s aggregate
High concurrency Many requests, strong scheduling and cache reuse ~18,000 tokens/s aggregate
Example: if the system serves 64 concurrent requests and each averages 280 generated tokens, the cluster-level throughput can be very high even when each individual request feels conversational. That is the core benefit of continuous batching and paged KV management. [web:18][web:8]

6. Concrete Serving Math

Here is a simple way to think about memory headroom on a 141 GB H200. If weights take about 10.5 GB at 4-bit and KV cache for a long 128k request takes about 4.5 GB, then the remaining memory must cover runtime overhead and concurrency. [web:12][web:11]

# Rough serving budget example GPU VRAM: 141 GB Weights (INT4): ~10.5 GB KV cache for 128k prompt: ~4.5 GB Overhead + workspace + fragmentation: depends on engine Effective headroom: still large enough for multi-request serving

A key observation is that long prompts do not automatically imply low concurrency. If many requests share a prefix, prefix caching can save significant memory and increase throughput further. [web:8][web:3]
